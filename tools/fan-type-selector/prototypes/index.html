Result Layout Concepts

These change how the same selection result is read, but keep the chart ideas secondary.

Prototype A: Result Subtabs

Keeps the top recommendation shallow, then pushes the rest of the result into single-column sub-tabs inside the Results area: Overview, Hardware, Alternatives, and Technical Basis. This is the lowest-risk production refactor.

  • Best when the user still wants conventional in-page navigation.
  • Reduces scroll by only showing one result slice at a time.
  • Still feels like a normal engineering tool, not a custom app shell.
Open Prototype A

Prototype B: Drawer Workbench

Treats the result as a shallow decision dashboard. The main page stays short and calm; deeper packaging, alternatives, and technical basis open in a right-hand drawer.

  • Most aggressive at killing scroll.
  • Strongest for “check one thing, then close it” workflows.
  • More custom interaction, so higher implementation cost.
Open Prototype B

Prototype C: Compact Accordion

Keeps everything in one column, but turns the results into a tight accordion with a sticky section rail. This keeps the mental model simple while reducing visible height.

  • Best if the team wants a conservative UI move.
  • Feels close to the current tool, but much calmer.
  • Still allows a full-page reading flow when needed.
Open Prototype C

Prototype D: Verdict Then Evidence

The decision is the opening sentence. A numbered funnel shows how four types narrowed to one, with each step expanding to show the reasoning. Numbers appear only where they support a specific step, not in a detached grid.

  • Makes the "why" impossible to miss — it's the first thing on the page.
  • Linear funnel replaces nested tabs entirely.
  • Each step is self-contained: question, answer, optional evidence.
Open Prototype D

Prototype E: Score Waterfall

All four types visible at once with stacked horizontal bars showing what each scored and why. Click any row to expand its hardware details and trade-offs. The visual gap between bars IS the explanation.

  • Score components are immediately visual — no table hunting.
  • Comparison is always on screen; details expand inline.
  • Works well for users who think in relative terms ("how close is #2?").
Open Prototype E

Prototype F: Guided Narrative

Reads top-to-bottom like a decision memo with a timeline spine. Each step shows its conclusion on the fold line; expand to see the supporting evidence. No tabs, no metric grid — just the story of how the decision was made.

  • Most natural reading order — mirrors how an engineer would explain it verbally.
  • Ranked comparison cards with hardware numbers inline, not in a separate view.
  • Callout boxes flag the one or two things that could overturn the recommendation.
Open Prototype F

Family Map Concepts

These are direct answers to the current Family Map weakness. The emphasis is on showing where the bounds come from and separating territory definition from temporary zoom level.

Prototype G: Explicit Bounds Bands

Replaces the irregular family polygons with declared comfort and fade bands. The user can read the flow and pressure ranges directly instead of inferring them from a freehand outline.

  • Strongest answer to “what defines the bounds?”
  • Turns the map into an inspectable data model, not hidden vertices.
  • Most production-friendly if we want the chart to look cleaner fast.
Open Prototype G

Prototype H: Focus Window + Inset

Keeps a global family field, but adds a visible viewport mode. The inset shows the full screening space while the main chart can zoom around the current duty point.

  • Strongest answer to “why are the axis limits here?”
  • Separates permanent territory bounds from temporary viewing focus.
  • Best if we want to preserve the map feel without arbitrary-looking extents.
Open Prototype H
